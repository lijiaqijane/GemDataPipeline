## 完整搭建流程（从零到跑通）

> 以下步骤记录从克隆仓库到成功运行 AgentGEM 项目的完整过程。
> 环境：Windows 11 + Docker Desktop + WSL2 (Ubuntu-22.04) + Python

---

### Step 1：克隆项目

```powershell
# 在 Windows 中克隆
cd D:\Project
git clone https://github.com/lijiaqijane/GemDataPipeline.git
cd D:\Project\GemDataPipeline\GemDataPipeline
```

---

### Step 2：检查本机环境

```powershell
# 检查 Python 版本（需要 >= 3.10）
python --version

# 检查 Docker 是否安装
docker --version
docker compose version

# 检查是否有 NVIDIA GPU（用于 FAISS GPU 加速）
nvidia-smi
```

> 本机环境：Python 3.13.11、Docker 29.1.3、RTX 3090（24GB），CUDA Driver 12.6

---

### Step 4：创建 Python 虚拟环境并安装依赖

```powershell
# 在项目根目录创建虚拟环境
cd D:\Project\GemDataPipeline\GemDataPipeline
python -m venv .venv

# 激活环境（Windows）
.venv\Scripts\Activate.ps1

# 升级 pip
python -m pip install --upgrade pip

# 安装项目基础依赖
pip install -e .
# 如果需要搜索功能（faiss），安装完整依赖
pip install -e ".[full]"

# 也可安装 faiss-cpu（替代 faiss-gpu-cu12，因为后者不支持 Windows）
pip install faiss-cpu

# 验证安装
agent_gem --help
```

> ⚠️ **为什么不用 faiss-gpu-cu12？**
> 该包只有 Linux wheel（manylinux），不支持 Windows。
> 即使通过 WSL 运行，也用 faiss-cpu 替代，不影响其他 Agent 功能。
>
> ⚠️ **为什么不用 conda？**
> 本机未安装 conda，使用 venv 替代。`run.sh` 后续已修改为支持 venv。

### Step 7：进入 WSL 并激活环境

```bash
# 从 PowerShell 进入 WSL
wsl -d Ubuntu-22.04

# 进入项目目录
cd /mnt/d/Project/GemDataPipeline/GemDataPipeline
```

> ⚠️ **为什么在 WSL 中运行？**
> 项目代码通过 Serper API（Google 搜索）和 FireCrawl 抓取网页数据，
> 这些操作在 WSL 中网络连通性更好。同时 `run.sh` 是 bash 脚本，WSL 可直接运行。

### Step 8：设置环境变量

`run.sh` 中已包含默认值，核心变量如下：

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `VOLCANO_API_KEY` | `f5166447...` | DeepSeek API key（火山引擎） |

| `FIRECRAWL_API_URL` | `http://localhost:3102` | FireCrawl 服务地址 |
| `SANDBOX_FUSION_URL` | `http://localhost:8080` | SandboxFusion 服务地址 |

---

### Step 9：测试运行（单任务）

```bash
cd /mnt/d/Project/GemDataPipeline/GemDataPipeline
source .venv/bin/activate

# 跑 1 个类别、1 个任务、1 轮
NUM_CATEGORIES=1 CATEGORY="plan a travel itinerary" NUM_TASKS=1 ROUNDS=1 MAX_TOKENS=5000 ./run.sh
```

运行流程：
1. **Step 1** — 分析数据源
2. **Step 2** — 搜索 + FireCrawl 抓取网页（10 个数据文件）
3. **Step 3** — 综合任务工具（LLM 分析每个数据文件）
4. **Step 4** — 生成任务（描述 + 答案格式 + 参考代码 + 验证代码）
5. **Step 5** — 验证（LLM 执行验证代码，最多 5 轮修复）

输出示例：
```
Synthesized X task(s) across 1 category/categories:
- [difficulty] task_title: description
```

> 如果验证 5 轮后仍未通过，会输出 `Synthesized 0 task(s)`，这是 LLM 答案与数据不完全匹配导致的，
> 增加 `MAX_VALIDATION_ROUNDS=10` 或加 `VALIDATE=0` 跳过验证即可。

## 代码修改

### data_pipeline.py — extract 失败降级
`agent_gem/agents/general_agent/data_pipeline.py` 中 `_process_firecrawl_url` 方法：
当 FireCrawl extract（AI 提取）失败时，自动保存 scrape 得到的原始 markdown 内容，
避免因 extract 不兼容 DeepSeek 导致数据丢失。

### run.sh — 支持 .venv_wsl
在 Python 查找路径中增加了 `.venv_wsl/bin/python`，确保 venv 虚拟环境能被识别。

### FireCrawl DeepSeek 补丁（firecrawl/）
适配 deepseek 模型的特殊输出格式，增强了 JSON 解析的鲁棒性，
修复了 Docker 镜像中缺失的 `OPENAI_MODEL_MODE` 判断逻辑，
确保在容器环境中正确使用 Chat Completions API 而非 Responses API。

**持久化方式**：`firecrawl-copy/docker/patch.sh` + `helpers.js` 在容器启动时自动运行修补。
`docker-compose.yaml` 通过 volume 挂载 `docker/` 目录，
启动命令改为 `sh /docker/patch.sh && node dist/src/harness.js --start-docker`，容器重启后自动生效。

修改的源码文件：
- `firecrawl-copy/apps/api/src/lib/generic-ai.ts`
- `firecrawl-copy/apps/api/src/lib/extract/fire-0/llmExtract-f0.ts`
- `firecrawl-copy/apps/api/src/scraper/scrapeURL/transformers/llmExtract.ts`

## 常见问题

| 问题 | 原因 | 解决 |
|------|------|------|
| `ModuleNotFoundError: No module named 'coloredlogs'` | 用的是系统 Python，不是 venv | 运行 `source .venv/bin/activate` 激活环境 |
| `sandbox-fusion` 状态为 `Exited` | 内部 FastAPI 版本兼容问题 | `docker rm sandbox-fusion` 然后重新 `docker run` |
| FireCrawl 容器不存在 | 未启动 | 执行 `docker compose -p firecrawl --env-file .env.instance up -d` |
| 搜索返回 0 条结果 | Serper API key 无效或额度用完 | 检查 `echo $SERPER_API_KEY`，去 serper.dev 注册新 key |
| 验证一直失败（5 轮） | LLM 生成的答案与数据不完全匹配 | 增加 `MAX_VALIDATION_ROUNDS=10` 或加 `VALIDATE=0` 跳过验证 |
| `run.sh` 报错 `$'\r': command not found` | 文件是 Windows 换行符 | 在 WSL 中运行 `sed -i 's/\r$//' run.sh` 转换 |
